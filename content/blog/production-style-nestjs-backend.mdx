# How to Set Up a Production Style NestJS Backend

> This guide explains a setup model for backends that need to behave predictably from the moment they start. The same approach appears in the [Marketplace Service project](/projects/marketplace-service), where bootstrap, validation, logging, request correlation, and health checks are treated as part of the product rather than background plumbing.

## The Core Problem

When building a production grade backend, functional correctness is only one part of the job. Operational behavior matters just as much, because the service has to start cleanly, fail safely, expose useful signals, and remain understandable once it is deployed and carrying real traffic.

The endpoints may work, but the runtime contract can still be loose:

- **No fail-fast on bad config** — the app starts even when environment configuration is incomplete.
- **Inconsistent logging** — logs are inconsistent or split across multiple formats.
- **Unpredictable error shapes** — errors return different response shapes depending on where they happen.
- **No request correlation** — requests cannot be traced cleanly across logs.
- **Shallow health checks** — health endpoints do not clearly distinguish "process is up" from "service is ready."

This usually does not hurt during the first few local runs. It becomes a real problem once the service is deployed, restarted, debugged under pressure, or handed to someone else.

So the goal is not only to make the API work. The broader goal is to make the whole service predictable across startup, request handling, failure paths, debugging, deployment health, and runtime operations.

## Why Setup Matters Before Features

Production style setup is not a final polishing step. It defines the environment in which every feature runs.

If configuration is invalid, the process should fail early.

If a request fails, the client should still get a consistent error shape.

If something goes wrong in production, logs should already contain the request ID, method, path, status code, and failure classification.

> That means bootstrap behavior is part of the service contract, not just framework wiring.

## The Setup Model

The model is simple:

1. Validate environment configuration before the app starts listening.
2. Initialize one structured logging pipeline.
3. Attach a request ID before controllers run.
4. Apply global request validation once.
5. Normalize error responses centrally.
6. Expose separate liveness and readiness checks.

That creates one consistent request lifecycle instead of many local conventions.

## 1. Fail Fast on Invalid Environment Configuration

The safest startup behavior is to reject invalid config before the process begins accepting traffic.

That means two things:

1. Apply sensible defaults where appropriate.
2. Throw immediately when required values are missing or inconsistent.

Here is the kind of validation setup this article is referring to:

<CodeToggle
  title="Show TypeScript snippet: environment validation"
  language="typescript"
  code={`
class EnvironmentVariables {
    @IsInt()
    @Min(1)
    @Max(65535)
    PORT!: number;

    @IsString()
    DB_HOST!: string;

    @IsString()
    DB_USERNAME!: string;

    @IsString()
    DB_PASSWORD!: string;

    @IsString()
    DB_NAME!: string;

    @IsString()
    ADMIN_API_KEY!: string;
}

export const validateEnv = (
    config: Record<string, unknown>,
): Record<string, unknown> => {
    const configWithDefaults = {
        PORT: config.PORT ?? "3000",
        NODE_ENV: config.NODE_ENV ?? "development",
        LOG_LEVEL: config.LOG_LEVEL ?? "info",
        ...config,
    };

    const validatedConfig = plainToInstance(
        EnvironmentVariables,
        configWithDefaults,
        { enableImplicitConversion: true },
    );

    const errors = validateSync(validatedConfig, {
        skipMissingProperties: false,
    });

    if (errors.length > 0) {
        throw new Error(errors.toString());
    }

    return validatedConfig;
};
  `}
/>

> This approach is valuable because invalid runtime state is usually harder to debug than a startup crash. A clean failure during bootstrap is often the safest possible outcome.

## 2. Build One Shared Request Pipeline

Once configuration is valid, the next goal is to make every request pass through the same technical flow.

In NestJS, a practical way to handle that is to put the logic in a single `configureApp` function:

<CodeToggle
  title="Show TypeScript snippet: shared app pipeline"
  language="typescript"
  code={`
export const configureApp = <T extends INestApplication>(app: T): T => {
    // Pull one shared logger from the app container so every layer uses the same format.
    const structuredLogger = app.get<LoggerService>(WINSTON_MODULE_NEST_PROVIDER);

    // Register the common runtime behavior before feature routes start serving traffic.
    app.useLogger(structuredLogger);
    app.enableVersioning({
        type: VersioningType.URI,
    });

    // Add request correlation early so logs and errors can all reuse the same request ID.
    app.use(requestIdMiddleware);

    // Enforce one validation policy for every incoming request.
    app.useGlobalPipes(
        new ValidationPipe({
            whitelist: true,
            forbidNonWhitelisted: true,
            transform: true,
        }),
    );

    // Capture request completion and failures through shared infrastructure.
    app.useGlobalInterceptors(new RequestLoggingInterceptor(structuredLogger));
    app.useGlobalFilters(new AppExceptionFilter(structuredLogger));

    // Let the application respond cleanly when the environment sends shutdown signals.
    app.enableShutdownHooks();

    return app;
};
  `}
/>

This is useful because it keeps cross cutting behavior in one place:

1. request correlation
2. validation
3. request logging
4. exception normalization
5. graceful shutdown behavior

That is much easier to reason about than scattering those concerns across modules and controllers.

## 3. Add Request IDs at the Edge

Request IDs are one of the simplest improvements with one of the biggest debugging payoffs.

The main idea is:

1. preserve an incoming request ID if a caller already provided one
2. generate one if it is missing
3. attach it to both the request context and response headers

That looks like this:

<CodeToggle
  title="Show TypeScript snippet: request ID middleware"
  language="typescript"
  code={`
export const requestIdMiddleware = (
    request: RequestWithRequestId,
    response: Response,
    next: NextFunction,
): void => {
    // Reuse an incoming ID if an upstream system already created one.
    const incomingRequestId = request.header(REQUEST_ID_HEADER);

    // Otherwise create a fresh ID for this request.
    const requestId = incomingRequestId || randomUUID();

    // Store the ID on the request so downstream code and logs can reuse it.
    request.requestId = requestId;

    // Return the same ID in the response headers for easier debugging.
    response.setHeader(REQUEST_ID_HEADER, requestId);

    // Continue to the next middleware or controller.
    next();
};
  `}
/>

Once this exists, every completion log and error log can include the same request identifier. That makes debugging much faster because one failing request can be followed across middleware, controller, service, and exception logs.

## 4. Enforce Request Shape Centrally

Validation is stronger when it is consistent.

Instead of validating in ad hoc controller code, the request boundary should reject malformed input automatically.

The important settings here are:

1. `whitelist: true`
2. `forbidNonWhitelisted: true`
3. `transform: true`

That combination does three helpful things:

1. strips unknown properties only when the contract allows it
2. rejects unexpected fields instead of silently accepting them
3. converts DTO bound values into the expected runtime shape

This reduces controller noise and prevents request handling from turning into defensive parsing code everywhere.

## 5. Normalize Error Responses in One Place

Clients should not have to guess whether an error came from validation, a service rule, or an unhandled exception.

A global exception filter gives one response shape and one structured logging path:

<CodeToggle
  title="Show TypeScript snippet: global exception filter"
  language="typescript"
  code={`
@Catch()
export class AppExceptionFilter implements ExceptionFilter {
    constructor(private readonly logger: LoggerService) {}

    catch(exception: unknown, host: ArgumentsHost): void {
        // Get access to the HTTP request and response involved in this failure.
        const context = host.switchToHttp();
        const request = context.getRequest<RequestWithRequestId>();
        const response = context.getResponse<Response>();
        const timestamp = new Date().toISOString();

        // Use the framework status when available, otherwise fall back to 500.
        const statusCode =
            exception instanceof HttpException
                ? exception.getStatus()
                : HttpStatus.INTERNAL_SERVER_ERROR;

        // Write one structured error event with the important request context attached.
        this.logger.warn({
            event: "http.request.error",
            requestId: request.requestId,
            method: request.method,
            path: request.originalUrl || request.url,
            statusCode,
        });

        // Return one predictable error shape to the client.
        response.status(statusCode).json({
            statusCode,
            requestId: request.requestId,
            timestamp,
        });
    }
}
  `}
/>

> The point is not only cleaner code. The point is stable API behavior under failure.

## 6. Separate Liveness from Readiness

> A backend can be alive but not actually ready to serve traffic.

That is why it is best to expose two health checks:

1. liveness: the process is running
2. readiness: the process can reach the dependencies it needs, such as the database, cache, or email service

A simple readiness check often looks like this:

<CodeToggle
  title="Show TypeScript snippet: readiness check"
  language="typescript"
  code={`
async getReadiness() {
    // First confirm the main database connection was initialized.
    if (!this.dataSource.isInitialized) {
        throw new ServiceUnavailableException({
            message: "Database connection is not initialized.",
            errorCode: "DATABASE_NOT_READY",
        });
    }

    // Then run a lightweight query to prove the dependency is actually reachable.
    await this.dataSource.query("SELECT 1");

    // Return a small status payload that deployment systems can inspect.
    return {
        status: "ok",
        checks: {
            database: "up",
        },
        timestamp: new Date().toISOString(),
    };
}
  `}
/>

This distinction matters during deployment and restarts. A process that has booted but cannot reach the database should not be treated as ready.

## Putting the Flow Together

The full request lifecycle becomes:

1. bootstrap validates config
2. app initializes one logging pipeline
3. request enters with a request ID
4. DTO validation runs globally
5. controller and service logic execute
6. request completion is logged with structured metadata
7. failures are normalized by the global exception filter
8. health endpoints reflect runtime and dependency state separately

That is a much stronger foundation than "Nest app starts and controllers exist."

## Practical Tradeoffs

- **Centralized setup vs drift** — Centralized setup improves consistency, but it requires discipline to keep cross-cutting concerns in the shared pipeline instead of drifting back into feature modules.
- **Strict startup vs flexibility** — Fail-fast config validation makes startup stricter, but that is usually better than discovering bad config through partial runtime failure.
- **Structured logging vs payload discipline** — Structured logging is more useful operationally, but it takes effort to keep payloads consistent and free of sensitive values.
- **Real dependency health vs shallow checks** — Readiness checks improve deployment safety, but they must reflect real dependency health instead of shallow process checks.

## How This Applies to My Marketplace Service

In the [Marketplace Service project](/projects/marketplace-service), this setup model establishes the technical behavior of the application before any business endpoint starts serving traffic.

Environment variables are validated up front. The application uses one structured logger, not a mix of console output and framework defaults. Request IDs are attached before controllers run, then reused in request completion logs and error logs. Validation rules are enforced globally, which keeps the controllers thin. Error responses are normalized by one exception filter, and the service exposes both liveness and readiness endpoints so runtime health is clearer during deployment and debugging.

That combination makes the service easier to operate, not just easier to code.
