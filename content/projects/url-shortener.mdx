# Trimly URL Shortener


## Core platform capabilities

- **Short link creation and resolution:** Accepts source URLs and returns short codes with stable redirect behavior.
- **High-speed redirect read path:** Uses Redis cache-aside for fast lookups before database fallback.
- **Negative caching for misses:** Stores short-lived miss keys to reduce repeated database work for invalid or expired codes.
- **Safe runtime control:** Supports Redis enable/disable toggle for controlled rollout and instant rollback behavior.

## Engineering practices and concepts

I treated Trimly as a backend reliability exercise where speed improvements must never break user-facing correctness.

### Redirect performance strategy

- Kept the redirect hot path minimal with clear command order: positive key check, miss key check, then DB fallback.
- Used compact cache payloads and bounded Redis operation timeouts to reduce overhead in critical reads.
- Maintained DB as source of truth while using cache for acceleration.

### Cache safety and data correctness

- Used versioned Redis key prefixes for controlled evolution of cache shape.
- Applied separate TTL strategy for positive hits and miss keys.
- Ensured write flows invalidate or refresh related cache keys to avoid stale redirects.

If you want a step-by-step explanation of how this caching flow works in practice, read [URL Shortener Caching Guide](/blog/url-shortener-caching).

### Resilience and failure handling

- Added startup fail-fast checks when Redis is enabled, including connectivity validation.
- Ensured graceful fallback to DB-only path when Redis is disabled or unavailable.
- Preserved stable HTTP behavior across all paths (`302` for valid redirects, `404` for missing or expired links).

For create-path reliability under retries, see [Idempotency in API Design](/blog/idempotency-in-api-design).

### Observability and load validation

- Tracked cache hits, negative-cache hits, timeout/error counts, and fallback-to-DB frequency.
- Measured latency and throughput with repeatable load-test workloads.
- Verified Redis-on vs Redis-off performance to confirm real gains instead of assumptions.
